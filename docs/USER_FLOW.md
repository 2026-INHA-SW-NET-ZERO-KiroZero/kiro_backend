# 냉장고 반상회 MVP 사용자 플로우

## 1. 전체 흐름 요약

냉장고 반상회 MVP는 seed 데이터로 미리 생성된 조리실습실 장소/시간/화구 스테이션 슬롯을 기반으로 진행된다.

인하대 조리실습실은 화구 6개를 `A`, `B`, `C`, `D`, `E`, `F` 스테이션으로 나누고, 각 스테이션을 4명 정원의 하나의 방으로 운영한다. 시간대는 우선 `16:00~18:00`, `18:00~20:00` 두 타임으로 나누어 하루 최대 12개 슬롯을 생성할 수 있다.

사용자는 가능한 슬롯에 참여하고, 소진하고 싶은 재료를 식재료 마스터에서 검색해 선택한다.

4명이 모이면 시스템은 참여자들의 재료를 하나의 공유 재료 풀로 합산한다.

그 후 백엔드는 추가구매 가능자를 확인하고, Bedrock이 내부 후보로 일반식 후보 3개와 저탄소 후보 3개를 생성하면 백엔드가 JSON 스키마와 최소 정책을 확인한 뒤 일반식 2개와 저탄소 2개를 A/B/C/D로 사용자에게 보여준다.

참여자들은 A/B/C/D/E 다섯 선택지로 메뉴를 투표한다. A/B는 일반식 후보, C/D는 저탄소 후보, E는 후보가 마음에 들지 않는 경우다. 하나의 메뉴가 과반을 넘으면 세션이 확정된다.

당일 조리 후 실제 소진량을 기록한다. 사용자 결과 요약은 RDS 원자료로 계산하고, 같은 계산 결과를 DynamoDB `campus_sustainability_reports`에 완료 세션 성과 로그로 저장한다. 학교·지자체·스마트캠퍼스 성과 대시보드는 이 DynamoDB 로그를 읽는다.

## 2. 핵심 플로우

```text
사용자가 로컬 회원가입
→ 이메일 도메인 검사
→ 닉네임/요리 숙련도/알레르기 등록
→ 데모 seed로 조리실 장소/시간 슬롯 생성
→ 사용자가 슬롯 목록 확인
→ 사용자가 추가구매 가능 여부와 소진할 재료 1개 이상을 입력해 공개 슬롯 참여
→ 4명 모집 완료
→ 시스템이 공유 재료 풀 생성
→ 백엔드가 추가구매 가능자 여부 확인
→ Bedrock이 내부 후보 6개 생성
→ 백엔드가 JSON 스키마와 최소 정책 확인
→ 일반식 2개와 저탄소 2개를 후보 A/B/C/D로 표시
→ 구매 가능자가 있으면 추가구매 담당 자동 배정
→ 참여자 A/B/C/D/E 메뉴 투표
→ 세션 확정
→ Bedrock이 단계별 협업 조리 플랜 생성
→ 조리 진행
→ 소진량 기록
→ 환급액 산정
→ 세션 결과 요약 반영
→ DynamoDB 성과 로그 저장
→ 스마트캠퍼스 성과 대시보드 반영
```

## 3. 데모 seed와 결과 요약 플로우

### 3.1 운영자 UI 제외

MVP에서는 별도 운영자 로그인이나 관리자 콘솔을 만들지 않는다.

운영자가 하려던 역할은 대부분 해커톤 데모에서 seed 데이터로 대체할 수 있다.

### 3.2 장소/시간 슬롯 seed 생성

조리 가능 장소, 시간, 화구 스테이션은 seed 데이터 또는 데모 seed 생성 API로 미리 등록한다.

슬롯은 메뉴 방이 아니라 장소/시간 기반 조리 세션이다.

MVP 운영 단위는 다음과 같다.

- 조리실습실 화구 6개를 `A`~`F` 스테이션으로 구분한다.
- 각 스테이션은 4명 정원의 독립 슬롯이다.
- 시간대는 `16:00~18:00`, `18:00~20:00` 두 타임으로 나눈다.
- 예를 들어 `18:00~20:00 / 스테이션 C`는 하나의 방이다.

seed에 포함되는 정보는 다음과 같다.

- 장소명
- 스테이션
- 날짜
- 시작 시간
- 종료 시간
- 정원
- 조리 가능 시간
- 사용 가능 기기
- 공용 조미료 키트
- 허용 재료군
- 제한 재료군
- 모집 마감 시간
- 세션 설명

예시:

```text
장소: 인하대 조리실습실
스테이션: C
시간: 2026-06-29 18:00~20:00
정원: 4명
조리 가능 시간: 90분
사용 가능 기기: 인덕션, 프라이팬, 냄비, 전자레인지
공용 키트: 식용유, 소금, 후추, 간장, 고추장, 참기름
제한: 집에서 가져온 생고기, 해산물, 조리 후 남은 음식
```

데모 seed 생성 대상은 다음과 같다. 식재료 마스터 100개는 코드 enum이 아니라 SQL seed 파일을 기준으로 RDS에 적재한다.

- [003_demo_slots_seed.sql](/Users/chataehun/Documents/아이디어/kiro_backend/src/main/resources/db/seed/003_demo_slots_seed.sql): 장소/시간/스테이션 슬롯
- [001_ingredient_master_seed.sql](/Users/chataehun/Documents/아이디어/kiro_backend/src/main/resources/db/seed/001_ingredient_master_seed.sql): 100개 식재료 마스터
- [004_demo_users_seed.sql](/Users/chataehun/Documents/아이디어/kiro_backend/src/main/resources/db/seed/004_demo_users_seed.sql): 데모 사용자
- 예시 세션 데이터

### 3.4 세션 결과 요약

완료된 세션의 성과는 학생 앱 안의 세션 결과 요약으로 먼저 보여준다. 사용자 결과 요약은 RDS 원자료를 계산해 만들고, 같은 계산 결과는 DynamoDB `campus_sustainability_reports`에 완료 세션 성과 로그로 저장한다.

결과 요약에서 확인할 정보:

- 세션 재료 소진량
- 예상 음식물쓰레기 감소량
- 버려질 뻔한 식재료의 추정 탄소 가치
- 선택 메뉴의 저탄소 여부
- 재료 소진율
- 완성 음식 소비율
- 환급 점수와 환급액
- 완성 음식 사진/식후 사진

학교·지자체·스마트캠퍼스 성과 대시보드는 HTML의 기존 운영자 대시보드 성격 화면을 이름과 역할만 바꿔 사용한다. 이 화면은 슬롯 생성이나 사용자 관리를 하지 않고, DynamoDB 성과 로그와 S3 사진 URL을 활용해 누적 성과를 보여준다.

## 4. 사용자 플로우

### 4.1 사용자 진입

사용자는 앱에 접속한다.

MVP에서는 OAuth를 사용하지 않고 로컬 회원가입을 사용한다.

회원가입 시 이메일 도메인이 `inha.ac.kr` 또는 `inha.edu`인지 검사한다. 도메인 검사는 학교 구성원 여부를 확인하기 위한 MVP 수준의 간단한 인증 장치다.

사용자는 앱에서 실명 대신 닉네임으로 표시된다.

단, 시스템 내부에서는 학교 인증 사용자로 관리된다.

회원가입 시 입력하는 정보는 다음과 같다.

- 이메일
- 비밀번호
- 닉네임
- 요리 숙련도
- 대표 알레르기 체크박스 및 기타 알레르기 내용

요리 숙련도는 `상`, `중`, `하` 3개 선택지로 둔다.

- 상: 조리에 익숙하고 메인 조리 가능
- 중: 간단한 조리와 보조 조리 가능
- 하: 초보, 손질/정리/보조 역할 선호

알레르기는 대표 알레르기 체크박스와 `없음`, `기타` 입력을 함께 제공한다. 대표 알레르기는 식재료 마스터의 알레르기 태그와 비교해 후보 검증에 사용한다.

### 4.2 슬롯 목록 조회

사용자는 열려 있는 조리 세션 슬롯을 확인한다.

방 카드에는 메뉴가 표시되지 않는다.

메뉴는 4명의 재료가 모인 후 생성된다.

방 카드 예시:

```text
6/29 저녁 냉장고 반상회
장소: 조리실습실
스테이션: C
시간: 18:00~20:00
현재 인원: 2/4명
공용 조미료 제공
가져올 재료: 검색 선택
메뉴는 4명의 재료가 모이면 추천됩니다.
```

### 4.3 슬롯 참여

사용자는 가능한 슬롯에 참여한다.

회원가입 시 이미 받은 정보는 슬롯 참여 시 다시 입력하지 않는다. 슬롯 참여 화면에서는 해당 세션에 필요한 정보만 받는다.

회원가입 시 보유한 사용자 기본 정보:

- 닉네임
- 요리 숙련도
- 알레르기

슬롯 참여 시 입력할 정보:

- 참여할 슬롯
- 추가구매 가능 여부
- 소진하고 싶은 재료
- 재료 선택
- 개수
- 실제 g 선택 입력

요리 숙련도 예시:

- 상
- 중
- 하

추가구매 가능 여부 예시:

- 불가능
- 가능하면 할 수 있음
- 세션 전에 마트/편의점에 들를 수 있음
- 신선 구매 재료도 구매 가능

MVP에서는 추가구매 가능 여부와 소진할 재료를 슬롯 참여 과정에서 함께 받는다. 재료를 1개 이상 선택하지 않으면 참여 버튼을 활성화하지 않는다.

### 4.4 재료 입력

사용자는 냉장고의 모든 재료를 입력하지 않는다.

사용자는 소진하고 싶은 재료를 식재료 마스터에서 검색해 선택한다.

다만 이 서비스는 냉장고 전체 재고 관리 앱이 아니므로, 사용자는 이번 세션에 실제로 가져가고 싶은 소진 대상 재료만 입력한다.

식재료 마스터는 사용자 검색/선택용 재료만 관리한다.

MVP에서는 자취생이 자주 쓰는 재료 100개를 [001_ingredient_master_seed.sql](/Users/chataehun/Documents/아이디어/kiro_backend/src/main/resources/db/seed/001_ingredient_master_seed.sql)로 먼저 등록한다. 1개당 추정 g, 알레르기 태그, 탄소계수는 공공데이터만으로 채우기 어렵기 때문에 팀이 직접 검토한 MVP 기준값으로 정의한다.

사용자 입력 화면에서는 식재료 마스터에 등록된 검색용 재료만 조회된다. 생고기, 생해산물처럼 사용자 지참을 받지 않는 고위험 재료는 식재료 마스터에 넣지 않는다. Bedrock이 일반식 후보에서 추가구매 품목을 제안할 때는 후보 JSON의 `purchase_items` 텍스트 필드로만 다룬다.

입력 항목은 다음과 같다.

- 재료 선택
- 개수
- 실제 g 선택 입력

예시:

```text
재료: 양배추
개수: 0.5개
추정 중량: 400g
```

사용자가 실제 무게를 알고 있는 경우에는 같은 재료 입력에 실제 g을 함께 넣을 수 있다.

```text
재료: 양배추
개수: 0.5개
실제 g: 350g
계산 기준 중량: 350g
```

### 4.5 재료 입력 제한

사용자의 남은 고위험 재료는 허용하지 않는다.

이 제한은 LLM에게만 맡기지 않는다. 사용자 입력 화면에는 애초에 검색 가능한 식재료 마스터만 노출하고, 백엔드는 Bedrock 후보 확인 단계에서 추가구매 품목의 최소 정책을 다시 확인한다.

금지 예시:

- 집에 남아 있던 생고기
- 집에 남아 있던 해산물
- 조리 후 남은 음식
- 보관 상태가 불명확한 유제품

이러한 재료가 필요한 경우에는 세션 직전 추가구매 품목으로만 조건부 허용한다.

처리 흐름은 다음과 같다.

```text
사용자가 식재료 마스터에서 재료 선택
→ 개수 입력
→ g을 알고 있으면 실제 g 선택 입력
→ 백엔드가 실제 g이 있으면 실제 g, 없으면 1개당 추정 g 기준으로 추정 중량 계산
→ 식재료 마스터 재료만 공유 재료 풀에 포함
→ 필터링된 재료 풀만 Bedrock에 전달
→ Bedrock 메뉴 후보의 추가구매 품목을 백엔드가 최소 정책 확인
```

예를 들어 `돼지고기`나 `새우`는 사용자 재료 입력 화면에서 검색되지 않는다. 다만 일반식 후보의 추가구매 품목으로는 조건부 제안될 수 있고, 이때 백엔드는 구매 가능자와 알레르기 충돌 여부를 검증한다.

### 4.6 4명 모집 완료

4명이 모이면 시스템은 세션을 다음 단계로 전환한다.

방 참여는 재료 1개 이상 입력과 동시에 완료된다. 따라서 4명이 모였다는 것은 각 참여자가 이미 최소 1개 이상의 재료를 제출했다는 뜻이다.

조건:

```text
참여자 수 = 4
알레르기 정보 입력 완료
추가구매 가능 여부 입력 완료
```

### 4.7 공유 재료 풀 생성

시스템은 4명의 재료를 하나의 공유 재료 풀로 합산한다.

예시:

```text
A님의 입력
- ingredient_id: 12 / 양배추 / count: 0.5 / knownGrams: 350 / estimatedGrams: 350
- ingredient_id: 21 / 대파 / count: 1 / knownGrams: null / estimatedGrams: 80

B님의 입력
- ingredient_id: 7 / 밥 / count: 1 / knownGrams: 210 / estimatedGrams: 210
- ingredient_id: 34 / 김치 / count: 1 / knownGrams: 150 / estimatedGrams: 150

C님의 입력
- ingredient_id: 3 / 양파 / count: 1 / knownGrams: null / estimatedGrams: 200
- ingredient_id: 41 / 계란 / count: 3 / knownGrams: null / estimatedGrams: 180

D님의 입력
- ingredient_id: 9 / 당근 / count: 1 / knownGrams: null / estimatedGrams: 120
- ingredient_id: 44 / 두부 / count: 1 / knownGrams: 300 / estimatedGrams: 300

공유 재료 풀
- ingredient_id: 12 / 양배추 / totalEstimatedGrams: 350
- ingredient_id: 21 / 대파 / totalEstimatedGrams: 80
- ingredient_id: 7 / 밥 / totalEstimatedGrams: 210
- ingredient_id: 34 / 김치 / totalEstimatedGrams: 150
- ingredient_id: 3 / 양파 / totalEstimatedGrams: 200
- ingredient_id: 41 / 계란 / totalEstimatedGrams: 180
- ingredient_id: 9 / 당근 / totalEstimatedGrams: 120
- ingredient_id: 44 / 두부 / totalEstimatedGrams: 300
```

사용자는 개수를 필수로 입력하고, 실제 g을 아는 경우에만 `knownGrams`를 추가 입력한다. 공유 재료 풀 화면과 Bedrock 입력에는 `estimatedGrams`를 기준으로 전달한다.

이 화면은 서비스의 핵심 장면이다.

### 4.8 메뉴 후보 생성

시스템은 Bedrock에 다음 정보를 전달한다.

- 공유 재료 풀
- 각 재료의 대략적 양
- 식재료별 추정 중량
- 공용 조미료 키트
- 조리 가능 기기
- 조리 가능 시간
- 알레르기
- 구매 가능자 여부
- 추가구매 가능자 수
- 추가구매 허용 여부
- 추가구매 비용 참고 상한
- 제한 재료 정책
- 과거 유사 세션 요약

Bedrock은 내부 후보 6개를 생성한다.

후보 구성은 다음과 같다.

- 일반식 후보 3개
- 저탄소 후보 3개

일반식 후보는 고기류나 어패류처럼 추가구매가 필요한 재료를 조건부로 포함할 수 있다. 사용자 화면에서는 이를 `일반식`으로 표현한다.

저탄소 후보는 고기류와 어패류를 제외하거나 최소화한 메뉴다. 기본적으로 남은 채소, 곡류, 두부, 계란, 버섯, 김치, 면류 등을 활용해 추가구매 부담과 탄소 부담을 낮추는 방향으로 생성한다.

단, 최종 정책 확인과 사용자 화면에 보여줄 4개 후보 선택은 백엔드가 수행한다.

백엔드는 JSON에 없는 `ingredient_id`, 허용되지 않는 `menu_type`, 파싱 실패, 알레르기 태그 누락처럼 검증할 수 없는 응답을 받으면 해당 후보를 비활성화하거나 Bedrock 재호출 대상으로 처리한다.

Bedrock이 담당하는 것은 다음과 같다.

- 메뉴 이름 생성
- 어떤 남은 재료를 사용할지 제안
- 공용 키트로 사용할 조미료 제안
- 필요한 경우 신선 구매 품목 제안
- 후보 비교용 조리 흐름 요약 생성
- 후보 비교용 역할군 요약 생성
- 추천 이유 문장 생성

Bedrock이 담당하지 않는 것은 다음과 같다.

- 고위험 재료 허용 여부 최종 판단
- 알레르기 충돌 최종 판단
- 환급 점수 계산
- 추가구매 담당 배정
- 세션 상태 변경
- 메뉴 확정

이 판단들은 모두 백엔드가 수행한다.

### 4.9 후보 검증과 메뉴 카드 표시

백엔드는 Bedrock이 생성한 6개 후보를 다음 기준으로 검증한다.

- 알레르기 충돌 여부
- 추가구매 가능자 존재 여부
- 추가구매 담당 자동 배정 가능 여부
- 남은 재료로 사용된 `ingredient_id`가 현재 세션 재료에 존재하는지 여부
- 조리 시간과 조리도구 적합성
- 공용 키트로 해결 가능한 정도
- 메뉴 유형
- 예상 소진율
- 예상 환급 가능성

추가구매 가능자가 없으면 백엔드는 Bedrock에 추가구매 없는 메뉴를 우선 요청하거나, 추가구매가 필요한 후보를 선택 불가 처리한다.

추가구매 가능자가 있으면 백엔드는 그 사람들 중 한 명을 무작위로 자동 배정한다. 이 배정은 전체 참여자 랜덤이 아니라, 사용자가 사전에 `추가구매 가능`이라고 입력한 사람들 안에서만 수행한다. 자동 배정은 별도 수락 단계가 아니라 메뉴 후보 화면에 바로 표시되는 결과값이다.

검증을 통과한 후보 중 사용자에게는 일반식 후보 2개와 저탄소 후보 2개만 표시한다. 이때 백엔드는 후보를 별도 테이블에 저장하지 않고, 요청 처리 중 메모리에서 간단한 선정 기준을 적용한 뒤 최종 4개 후보만 `slots.candidates_json`에 저장한다.

메뉴 후보 예시:

```text
후보 C. 김치 야채 볶음밥
유형: 저탄소
사용되는 남은 재료: 밥, 김치, 양파, 대파, 계란
공용 키트: 식용유, 간장, 참기름
추가구매: 없음
예상 소진율: 86%
예상 환급: 높음
추가구매비: 없음
```

```text
후보 A. 닭가슴살 야채 덮밥
유형: 일반식
사용되는 남은 재료: 양파, 양배추, 당근, 대파, 밥
공용 키트: 식용유, 간장, 후추
신선 구매 필요: 닭가슴살 400g
구매 담당: 민수 자동 배정
예상 소진율: 78%
예상 추가구매비: 6,000원
정산 방식: 참여자 자율 정산
예상 환급: 보통
```

참여자들이 보는 화면은 새로운 방이 아니라 현재 슬롯의 `메뉴 후보 화면`이다. 즉, 4명이 모인 기존 세션이 `MENU_PROPOSED` 상태로 바뀌고, 그 안에서 Bedrock이 만든 후보와 백엔드 검증 결과가 카드 형태로 표시된다. 참여자들은 이 화면에서 메뉴를 투표한다.

화면 형태는 자유 채팅방이 아니라 `세션 피드`에 가깝다. 채팅창처럼 위에서 아래로 흐르지만, 사용자가 아무 말이나 입력하는 구조가 아니라 시스템 카드와 버튼 중심으로 진행한다.

세션 피드 예시:

```text
[시스템] 4명의 재료가 모였습니다.
[재료 풀 카드] 양배추 350g, 대파 80g, 밥 210g, 김치 150g, 양파 200g, 계란 180g
[메뉴 후보 카드 A] 일반식 후보
[메뉴 후보 카드 B] 일반식 후보
[메뉴 후보 카드 C] 저탄소 후보
[메뉴 후보 카드 D] 저탄소 후보
[투표 버튼] A / B / C / D / E
[E 선택 시 입력창] 마음에 안 드는 이유를 직접 입력
```

이 구조는 채팅처럼 익숙하게 보이지만, 실제 기능은 메뉴 후보 확인, 투표, E 사유 입력, 결과 확인처럼 구조화되어 있다.

### 4.10 메뉴 투표

참여자는 메뉴 후보를 보고 A/B/C/D/E 중 하나에 투표한다.

MVP에서는 자유 채팅을 넣지 않는다.

구조화된 버튼으로 처리한다.

투표 선택지는 다음과 같다.

- A: 일반식 후보 1
- B: 일반식 후보 2
- C: 저탄소 후보 1
- D: 저탄소 후보 2
- E: 후보가 마음에 들지 않음

E를 선택한 사용자는 이유를 반드시 직접 입력한다. 이유 입력은 객관식으로 받지 않고 자유 텍스트로만 받는다.

예시:

```text
E 선택 이유:
고기 추가구매가 부담스럽고, 저탄소 후보는 너무 밍밍해 보여서 매콤한 면 요리 쪽으로 다시 추천받고 싶어요.
```

E 선택 이유는 백엔드가 저장한 뒤, 재추천이 필요할 때 4명의 직접 입력 텍스트를 모아 Bedrock에 다시 전달한다.

재추천 입력 예시:

```text
기존 공유 재료 풀
기존 후보 A/B/C/D
각 후보의 표시 라벨
E를 선택한 참여자들의 직접 입력 사유
알레르기
추가구매 가능 여부
저탄소 후보 2개 포함 조건
```

MVP에서는 재추천을 1회로 제한한다. 재추천 후에도 과반 메뉴가 없으면 세션을 확정하지 않고 `MENU_PROPOSED` 상태로 남긴다.

메뉴 확정 조건 예시:

```text
A/B/C/D 중 하나가 4명 중 3명 이상 선택
알레르기 충돌 없음
추가구매가 필요한 경우 구매 가능자 중 담당자 자동 배정 완료
```

A/B/C/D 중 하나가 과반을 넘으면 메뉴가 확정되고 세션이 확정된다.

C 또는 D 저탄소 후보가 확정되면 환급 산정에 반영한다. 세션 완료 후 실제 재료 소진율과 완성 음식 소비율까지 합산해 최종 환급액을 계산한다.

알레르기 충돌이 있으면 해당 메뉴는 확정할 수 없다.

### 4.11 세션 확정

메뉴가 확정되면 세션이 확정된다.

세션 확정 후 각 사용자에게 체크리스트가 제공된다.

체크리스트 예시:

```text
태훈님이 가져올 재료
- 양배추 350g
- 대파 80g

가져오지 않아도 되는 것
- 식용유
- 간장
- 소금
- 후추
- 참기름

공동 구매 품목
- 닭가슴살 400g
- 담당자: 민수
- 예상 구매비: 6,000원
- 정산: 앱 밖에서 참여자 자율 정산

공용비
- 기준 공용비: 2,000 크레딧
- 저탄소/소진 성과에 따른 환급 가능
```

### 4.12 단계별 협업 조리 플랜 생성

메뉴가 확정되면 백엔드는 확정 메뉴, 공유 재료 풀, 공용 키트, 조리 도구, 참여자 숙련도, 추가구매 담당 여부를 Bedrock에 다시 전달한다.

이 단계의 목적은 단순히 `재료 손질 담당`, `메인 조리 담당`처럼 큰 역할을 나누는 것이 아니다. Bedrock은 요리 시작부터 완성까지의 자세한 레시피를 만들고, 각 단계마다 4명의 참여자가 서로 겹치지 않게 무엇을 해야 하는지 배정한다.

Bedrock에 전달하는 정보:

- 확정 메뉴
- 사용할 남은 재료와 계획 사용량
- 공용 키트
- 추가구매 품목과 구매 담당자
- 조리 가능 시간
- 사용 가능한 조리 도구
- 참여자별 닉네임
- 참여자별 요리 숙련도
- 참여자별 알레르기
- 구매 담당 여부

Bedrock은 다음 형태의 세션 조리 플랜을 생성한다.

```json
{
  "cooking_plan_steps": [
    {
      "step_order": 1,
      "phase": "PREP",
      "title": "재료 분리와 세척",
      "estimated_minutes": 5,
      "instruction": "양배추, 양파, 당근을 씻고 조리대에 분리해 둡니다.",
      "used_ingredients": [
        {
          "ingredient_id": 12,
          "name": "양배추",
          "planned_use_grams": 280
        },
        {
          "ingredient_id": 3,
          "name": "양파",
          "planned_use_grams": 160
```

참여자별 작업은 짧은 명령문으로 끝내지 않는다. Bedrock은 각 작업마다 다음 정보를 함께 생성해야 한다.

- `task_purpose`: 왜 이 작업을 하는지
- `task_detail`: 어떤 재료를 얼마나, 어떤 방식으로 처리하는지
- `attention_points`: 초보자가 놓치기 쉬운 주의사항
- `display_instruction`: 사용자 화면에 그대로 보여줄 친절한 행동 지시문

예시:

```text
나쁜 예시:
양배추 280g을 채 썹니다.

좋은 예시:
볶을 때 양배추가 고르게 익고 물이 과하게 나오지 않도록, 양배추 280g의 물기를 털고 1cm 폭으로 채 썹니다. 두꺼운 심지는 조금 더 얇게 썰어주세요.
```

백엔드는 Bedrock이 만든 조리 플랜을 그대로 믿지 않고 다음을 검증한다.

- 모든 `participant_id`가 현재 슬롯 참여자인지
- 한 단계 안에서 한 참여자에게 서로 충돌하는 일이 배정되지 않았는지
- 추가구매 담당자에게 구매와 고난도 조리 역할이 과도하게 몰리지 않았는지
- 알레르기 보유자가 해당 알레르기 재료를 직접 다루도록 배정되지 않았는지
- 조리 시간이 슬롯 시간 안에 들어오는지
- 필수 재료와 공용 키트가 실제 후보 정보와 맞는지

검증을 통과한 조리 플랜은 세션에 종속되어 저장된다. 개인별 별도 방을 만들지 않고, 확정된 슬롯 안에서 같은 조리 플랜을 4명이 함께 본다.

### 4.13 조리 진행

참여자는 Bedrock이 만든 단계별 조리 플랜을 보면서 조리한다.

조리 진행 화면은 채팅이 아니라 타임라인형 가이드에 가깝다.

화면 예시:

```text
Step 2. 재료 손질 / 예상 10분

전체 지시:
채소는 볶음용으로 썰고, 닭가슴살은 한입 크기로 자릅니다.

태훈:
- 양배추를 1cm 폭으로 채 썰기

민수:
- 닭가슴살을 한입 크기로 자르고 후추 뿌리기

지우:
- 양파를 채 썰고 당근을 얇게 썰기

서연:
- 팬과 냄비 준비, 밥 데울 준비

주의:
생닭을 만진 도구는 채소 손질 도구와 섞지 않기
```

MVP에서는 실제 조리 과정을 앱이 실시간으로 추적하지 않는다. 체크박스나 타이머가 있으면 좋지만 필수는 아니다.

앱은 단계별 조리 순서, 참여자별 할 일, 예상 시간, 공용 키트 사용 항목, 안전 주의 문구를 제공한다. 같은 슬롯에 참여한 사용자는 같은 조리 플랜을 본다.

### 4.14 소진량 기록

조리 후 소진량 기록은 방에 종속된 하나의 제출물로 처리한다.

특정 대표 사용자를 미리 지정하지 않는다. 세션 참여자 중 한 명이 방 단위로 결과를 제출하고, 제출된 기록은 해당 세션의 결과로 저장된다. 다른 참여자는 같은 결과를 확인한다.

입력 방식은 선택지 기반이다.

각 재료마다 다음 버튼 중 하나를 선택한다.

- 0%
- 25%
- 50%
- 75%
- 100%

예시:

```text
양배추: 75%
대파: 100%
김치: 50%
밥: 100%
계란: 100%
```

완성 음식 소비율도 버튼 기반으로 입력한다.

- 0%
- 25%
- 50%
- 75%
- 100%

완성 음식 소비율은 재료 소진율과 함께 추정 음식물쓰레기 감소량, 식재료 탄소 가치, 환급 점수를 계산할 때 사용한다.

추가로 완성 음식 사진과 식후 사진을 업로드한다.

- 완성 음식 사진: 세션 결과 요약에서 오늘 만든 음식으로 표시
- 식후 사진: 실제 조리 행동과 음식물쓰레기 감축 증빙으로 저장

사진은 자동 잔반 판정이나 AI 채점에 사용하지 않는다. 소진율과 완성 음식 소비율은 사용자가 버튼으로 입력한 값을 기준으로 계산한다.

### 4.15 환급 산정

세션 결과가 제출되면 백엔드는 탄소중립 환급 점수를 계산한다.

환급 점수는 메뉴 후보를 정렬하기 위한 점수가 아니라, 공용비 환급액을 계산하기 위한 정산 기준이다.

예약 비용은 1인당 `2,000 크레딧`으로 고정한다.

- 저탄소 메뉴 선택: 40점
- 남은 재료 소진율: 30점
- 완성 음식 소비율: 30점

세 항목의 합은 100점이다.

```text
환급액 = 2,000 크레딧 * 50% * 환급 점수 / 100
최대 환급액 = 1,000 크레딧
```

예시:

```text
저탄소 메뉴 선택: 40점
남은 재료 소진율: 30점
완성 음식 소비율: 30점
환급 점수: 100점
환급액: 1,000 크레딧
```

### 4.16 세션 결과 요약

세션 완료 후 사용자는 세션 결과 요약을 확인한다.

이 화면은 RDS 원자료를 계산해 보여준다. 원자료는 `consumption_records`, `consumption_record_items`, `slots.selected_menu_json`, `ingredient_master`에 남고, 같은 계산 결과가 DynamoDB `campus_sustainability_reports`에 저장되어 있으면 조회 최적화용으로 사용할 수 있다.

예시:

```text
오늘 4명이 함께 소진한 재료: 약 1.4kg
내 재료 소진: 양배추 75%, 대파 100%
완성 음식 소비율: 100%
추가구매비: 앱 밖 자율 정산
기준 공용비: 2,000 크레딧
환급액: 1,000 크레딧
예상 음식물쓰레기 감소량: 1.4kg
버려질 뻔한 식재료의 추정 탄소 가치: X kgCO2e
완성 음식 사진: 표시
식후 사진: 증빙으로 저장
```

### 4.17 스마트캠퍼스 성과 대시보드

HTML의 기존 운영자 대시보드 성격 화면은 `스마트캠퍼스 성과 대시보드`로 이름과 역할을 바꾼다.

이 화면은 관리자 기능이 아니다.

- 슬롯 생성 없음
- 사용자 관리 없음
- 예약 관리 없음
- 완료 세션 성과 조회만 수행

성과 대시보드 지표:

- 누적 완료 세션 수
- 누적 참여자 수
- 누적 소진 식재료 kg
- 예상 음식물쓰레기 감소량
- 버려질 뻔한 식재료의 추정 탄소 가치
- 저탄소 메뉴 선택률
- 평균 재료 소진율
- 평균 완성 음식 소비율
- 총 환급액
- 완료 세션별 완성 음식/식후 사진 URL

조회 기준은 DynamoDB `campus_sustainability_reports`다.

### 4.18 내 신청/과거 모임

사용자는 현재 참여 중인 슬롯과 완료된 과거 모임을 확인할 수 있다.

진행 중 신청 내역:

- 날짜
- 장소
- 스테이션
- 시간
- 현재 상태
- 확정 메뉴명
- 내 재료 개수

과거 모임:

- 완료 날짜
- 확정 메뉴명
- 세션 총 소진량
- 추정 탄소 가치
- 환급액
- 결과 요약 화면 이동

이 화면은 새 테이블을 만들지 않고 `session_participants`, `slots`, `consumption_records`를 조합해 만든다.

### 4.19 프로필 수정

사용자는 가입 후 프로필을 수정할 수 있다.

수정 가능 항목:

- 닉네임
- 요리 숙련도
- 알레르기 태그

이메일은 학교 인증 기준이므로 수정하지 않는다. 못 먹는 음식은 MVP에서 받지 않는다.
- 완성 음식 사진/식후 사진 증빙

## 5. 주요 예외 상황

### 5.1 4명이 모이지 않는 경우

세션은 모집중 상태로 유지된다.

모집 마감 시간까지 4명이 모이지 않으면 데모에서는 해당 슬롯을 진행하지 않는다. MVP DB 상태값에는 별도 취소 상태를 두지 않는다.

MVP에서는 정원 4명을 기본 조건으로 둔다.

### 5.2 재료 없이 참여하려는 경우

사용자는 소진할 재료를 1개 이상 선택해야 방에 참여할 수 있다.

재료가 없는 상태에서는 참여 버튼을 비활성화하고, API도 `INGREDIENT_REQUIRED` 오류를 반환한다. 따라서 MVP에서는 참여 후 재료 입력 리마인드 상태를 만들지 않는다.

### 5.3 구매 담당자가 없는 경우

추가구매 가능자가 한 명도 없으면, 백엔드는 추가구매가 필요한 메뉴를 기본 후보로 만들지 않는다.

이 경우 Bedrock에는 `추가구매 없는 메뉴만 제안`하도록 요청한다. 만약 Bedrock 응답에 추가구매 품목이 포함되면, 백엔드가 해당 후보를 비활성화하거나 후보에서 제외한다.

### 5.4 알레르기 충돌

알레르기와 충돌하는 메뉴는 확정할 수 없다.

AI가 추천했더라도 백엔드에서 후보 제외 처리한다.

### 5.5 고위험 재료가 필요한 경우

사용자의 남은 고위험 재료는 허용하지 않는다.

해당 재료는 사용자 입력 화면에 노출하지 않는다. Bedrock이 추가구매 품목으로 제안한 경우에는 백엔드가 추가구매 가능 여부와 알레르기 충돌 여부를 검증한다.

## 6. MVP 플로우 우선순위

MVP에서 반드시 구현해야 하는 흐름은 다음과 같다.

```text
데모 seed로 슬롯 생성
→ 사용자 슬롯 참여
→ 재료 입력
→ 4인 재료 풀 생성
→ 메뉴 후보 생성
→ 후보 검증
→ 필요한 경우 구매 가능자 중 추가구매 담당 자동 배정
→ A/B/C/D/E 메뉴 투표
→ 단계별 협업 조리 플랜 생성
→ 소진량 기록
→ 환급액 산정
→ 세션 결과 요약
```

MVP에서 우선순위를 낮춰도 되는 흐름은 다음과 같다.

```text
실시간 채팅
푸시 알림
정교한 재료 대체 추천
학교 예약 시스템 연동
사진 기반 재료 인식
카카오 오픈채팅방 자동 생성/자동 초대/자동 운영
```
