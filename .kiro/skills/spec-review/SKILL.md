# 3. Spec Review

워크플로의 세 번째 단계. **코드를 쓰기 전에** 스펙을 검토해서 잘못된 결정이 코드로 굳어지지 않게 한다. 산출물은 "통과 / 수정 요청" 결정과 함께 구체적 코멘트.

## 1. 언제 쓰는가

- `spec-write` 가 끝난 직후 — 자기 검토
- 동료 스펙을 받아 1차 검토할 때
- "이 설계 들어가도 될까?" 류 질문

## 2. 핵심 원칙

- **머지 후가 아니라 머지 전, 코드 전이 아니라 스펙에서 잡는다.** 같은 결함을 코드에서 잡으면 비용이 10배.
- 스타일이 아니라 **구조 결함**을 본다 — 위치, 경계, 안티패턴.
- 명백한 문제가 없으면 **통과시킨다.** 패턴 더 화려하게 하라는 요구 금지.

## 3. 검토 순서

1. 목표가 한 문단으로 분명한가
2. API 계약이 안정적인가
3. 도메인 모델 위치가 맞는가
4. 상태 판단 위치가 맞는가
5. 트랜잭션/예외 경계가 안전한가
6. 적용/미적용 패턴 결정이 적절한가
7. 테스트 전략이 리스크를 덮는가

## 4. API 계약

- HTTP method 가 의미와 맞는가 (GET 으로 상태 변경 X)
- URL 이 리소스 중심인가
- request/response DTO 가 entity 를 직접 노출하지 않는가
- 상태 코드와 에러 envelope 가 일관적인가
- 페이지네이션이 필요한가
- breaking change 면 명시되었는가

## 5. 도메인 모델 위치 — 안티 체크리스트

| 잘못된 신호 | 의심해야 할 것 |
|---|---|
| DTO 파일 안에 도메인 enum | `entity/` 로 이동 |
| `enums/`, `types/` 디렉토리 | 도메인 옆으로 이동 |
| read-model 이 소비 도메인 패키지에 있음 | 데이터 출처 도메인으로 이동 |
| 이름이 소비 도메인 용어(`HomeItemSnapshot`) | 위치 재검토 |
| entity 가 Repository 를 호출 | 의존 방향 위반 — 인자로 받기 |

## 6. 상태 판단 위치 — 안티 체크리스트

- Service 가 도메인 필드 꺼내서 `if`/`when` 분기 → 도메인 메서드 후보
- 도메인 객체가 외부 데이터를 의존 → Service 로 분리해야 할 가능성
- 단일 도메인 판단 로직이 여러 Service 에 중복 → 도메인으로 합치기

## 7. 트랜잭션/예외 경계

- 다단 쓰기가 한 `@Transactional` 로 묶였는가
- 외부 API 호출이 트랜잭션 안에 있는가 (있으면 빼라)
- readOnly 표시가 적절히 있는가
- 비즈니스 실패는 custom exception 으로 분리되는가
- 광범위 `Exception` catch 금지 — 정당한 이유 없으면 안티패턴
- 예외 → HTTP 매핑이 일관적인가

## 8. 패턴 결정 — 안티 체크리스트

| 안티패턴 | 문제 | 대안 |
|---|---|---|
| Factory 남용 | 단순 생성이 복잡해짐 | `new` / DI |
| Strategy 남용 | 구현체 수만 늘어남 | 메서드 추출 |
| 깊은 Decorator 체인 | 실행 흐름 추적 어려움 | AOP / 명시적 조합 |
| Event 남용 | 핵심 비즈니스 흐름이 숨음 | 동기 호출 |
| Adapter 없음 | 외부 타입이 전역에 퍼짐 | 내부 인터페이스 정의 |
| Builder 를 단순 DTO 에 추가 | 의미 없는 boilerplate | 생성자 |
| 단일 구현체에 Strategy 인터페이스 | 가짜 추상화 | 구체 클래스 직접 사용 |

**기본 입장**: 패턴이 보이면 "지금 정말 필요한가?" 를 먼저 묻는다. "지금은 아니다" 라고 답할 줄 알아야 한다.

## 9. 테스트 전략

- 도메인 규칙/불변식이 unit 으로 덮이는가
- 컨트롤러 계약(상태 코드, 에러 envelope)이 slice 로 덮이는가
- 트랜잭션/JPA 매핑 리스크가 integration 으로 덮이는가
- 버그 수정인데 실패 테스트가 명시되지 않았다면 추가 요청

## 10. 출력 형식

```
# Spec Review — <기능 이름>

## Overall
- 결정: 통과 | 수정 요청
- 리스크: 낮음 | 중간 | 높음

## Critical (반드시 수정)
- <항목>: <문제> → <제안>

## Important (가능하면 수정)
- <항목>: <문제> → <제안>

## Minor (사소함, 선택)
- <항목>: <코멘트>

## Good
- <스펙에서 잘 결정된 부분>
```

## 11. 머지 결정 카테고리 (스펙 기준)

- **Critical**: 데이터 정합성/보안 영향, 잘못된 도메인 위치, 트랜잭션 안에 외부 호출, breaking API 미명시
- **Important**: 안티패턴 적용, 페이지네이션 누락, 중요한 테스트 누락, 모호한 예외 매핑
- **Minor**: 네이밍, 사소한 readability, 향후 리팩토링 후보

## 12. 하지 말 것

- 스펙에서 코드를 그리지 않는다 — 의사 코드는 짧게만
- 추상화/패턴을 더 넣으라는 요청을 기본값으로 하지 않는다
- 스타일 잔소리 (네이밍 외)
- "전반적으로 OK 같음" 류의 모호한 결론 — 통과면 통과, 수정이면 항목 지정

## 13. 한 줄 요약

**스펙 리뷰는 코드가 굳기 전 마지막 저비용 교정 지점. 구조 결함만 정확히 잡고 통과시킨다.**
