기능 개발 문서를 작성하고 Task를 분해하여 순차적으로 실행하는 스킬입니다.

## 절차

### 1단계: 개발 문서 생성

`$ARGUMENTS`를 분석하여 `docs/plans/YYYY-MM-DD-{기능명}.md` 파일을 생성하세요.

문서 형식:

```markdown
# {기능명}

## 상태: 🟡 진행중

## 개요
{기능에 대한 설명 - 사용자 입력 기반으로 작성}

## 기술 설계
- 관련 도메인: {도메인명}
- 필요한 엔티티: {목록}
- API 엔드포인트: {목록}
- 의존하는 다른 도메인: {있다면 명시}

## Tasks
- [ ] 🔴 Entity 클래스 생성
- [ ] 🔴 Repository 인터페이스 정의
- [ ] 🔴 DTO 정의 (요청/응답)
- [ ] 🔴 Service 비즈니스 로직 구현
- [ ] 🔴 Controller API 엔드포인트 구현
- [ ] 🔴 테스트 작성
- [ ] 🔴 ./gradlew harness 검증 통과

## 변경 이력
| 날짜 | Task | 상태 |
|---|---|---|
```

기능 특성에 따라 Task를 추가/수정할 수 있습니다. 위는 기본 템플릿입니다.

### 2단계: Task 순차 실행

Task를 위에서 아래로 하나씩 실행하세요.

각 Task를 시작할 때:
1. 개발 문서에서 해당 Task의 상태를 `🟡 진행중`으로 변경하고 `- [ ]`를 유지
2. 코드를 구현
3. 완료 시 상태를 `🟢 완료`로 변경하고 `- [x]`로 체크
4. 변경 이력 테이블에 기록 추가

### 3단계: 검증

모든 Task 완료 후:
1. `./gradlew harness` 실행
2. 통과하면 문서 상태를 `## 상태: 🟢 완료`로 변경
3. 실패하면 실패 원인을 수정하고 재실행

## 중요 규칙

- 반드시 `docs/specs/CONVENTIONS.md`의 코딩 컨벤션을 따르세요
- 반드시 `docs/specs/MAP.md`의 아키텍처 규칙을 따르세요
- Entity → Repository → DTO → Service → Controller → Test 순서를 지키세요
- 각 Task 완료 시 개발 문서를 반드시 업데이트하세요
